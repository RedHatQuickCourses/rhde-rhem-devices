Lab: Decommission and Delete Edge Devices

Estimated reading time: 5 minutes.

Objective

Make managed devices unmanaged and delete their device resources in RHEM.

Work in progress

Before you begin

You need a customer portal account with access to a Red Hat Enterprise Linux (RHEL) subscription. It could be a Red Hat Developer program free subscription.

You need a development machine running RHEL, which contains two test VMs already enrolled into your RHEM server by a previous lab.

You also need a server machine running RHEL, to which you also have unrestricted sudo access, which was already configured with RHEM.

If you are using the course classroom, your development machine is the workstation VM. Log in as the user student with password student.

Instructions

You will decommission and delete both managed devices you enrolled and configured in previous labs, and you will verify that the configurations set by RHEM remain on the devices after they are not managed anymore.

  1. Verify that you have the outcomes of previous labs. If you do not, go back and complete them, because you need their outcomes to proceed with this lab.

    While this activity does not depend on outcomes from all of the previous labs, it works better if you already perform some configuration of managed devices, after enrolling them.

    1. There is a clone of the course samples repository on your home directory.

      $ cd
$ ls rhde-rhem-samples
LICENSE  README.md  antora.yml  enroll  modules  webapp

    2. There are two local test VMs, named edge-1 and edge-2.

      $ virsh list --all

-------------------------
 -    edge-1   shut off
 -    edge-2   shut off

    3. If your test VMs are not started, because you stopped your classroom environment since completing the previous lab, start them now.

      $ virsh start edge-1
Domain 'edge-1' started
$ virsh start edge-2
Domain 'edge-2' started

    4. Enter the test-enroll directory, which contains the SSH keys used to provision and access your two test VMs.

      $ cd temp-enroll
$ ls
Containerfile  bib-iso.sh  config.toml  config.yaml  edge-key  edge-key.pub  output  rhem-enroll.iso  virt-install.sh

    5. Log in on RHEM, as an administrator, and verify that there are two managed devices, which correspond to the two test VMs.

      $ flightctl login -u admin -p redhat https://servera.lab.example.com:3443

$ flightctl get device -o wide

g7ombcqk46tqq458j9uju454441mf1lgt9qqqva8ti0fnvosm57g    edge-2  <none>  Unknown Unknown         Unknown         rhel=9.6,type=noapps,alias=edge-2

    6. Wait a few moments and repeat the latest command until both managed devices return as OnLine and UpToDate.

  2. Verify that you need full administrator rights to decomission managed devices.

    1. Open a web browser and visit https://servera.lab.example.com

    2. Log in as the student user with the student password. On the left pane, select the Devices page, and click the three vertical dots icon, to open the actions menu for any of the managed devices.

      GRAB SCREENSHOT

    3. Notice that there is no action do either decomission or delete a device, but only to edit device configurations and view device details.

      Also notice, to the top of page, bellow the device filters, a slider button to display decommissioned devices.

      GRAB SCREENSHOT

    4. Click the Show decommissioned devices button, and the page should show that there are no devices being decommissioned or already decommissioned.

      GRAB SCREENSHOT

    5. Click on the student user name, to the top right, to open the user menu and select Logout.

      GRAB SCREENSHOT

  3. Decomission the managed device which corresponds to your first test VM.

    1. Log in as the admin user with the redhat password. On the left pane, select the Devices page, and click the three vertical dots icon, to open the actions menu for the edge-2 managed device.

      GRAB SCREENSHOT

    2. Click Decommision device. Confirm that you really want to decommission the managed device by clicking on the red Decommision device button.

      GRAB SCREENSHOT

    3. The Devices page automatically switches to list decomissioned devices, and shows a status of Decomissioning for the device which has the UUID that used to belong to the edge-2 managed device. Notice that, when you decomission a managed device, it loses all its labels but it does NOT lose the configurations managed by RHEM.

      GRAB SCREENSHOT

    4. Click the UUID of the only device being decomissioned, which used to be the edge-2 managed device, and see that it still display the configurations which were set for the device.

      The device stays in that state on the assumption that some clean up procedure would be executed on it.

      GRAB SCREENSHOT

  4. Delete the decommissioned device.

    1. You could keep the device as being decomissioned for as long as you wish, but after a time you probably want to remove its records from the RHEM database.

      To remove the record for the device you just decomissioned, return to the Devices page and click again Show decommissioned devices. Then click the actions menu button for the only device listed and select Delete device.

      GRAB SCREENSHOT

    2. Confirm that you actually want to delete the device resource by clicking the red Delete device button. Now your Devices page should list no more decommissioned devices.

      GRAB SCREENSHOT

  5. Verify that decomissioned devices are not cleaned up.

    1. Switch to a terminal and enter the temp-enroll directory, which contains the SSH keys for acessing your test VMs.

      Notice that it still contains the message of the day that you configured by using RHEM.

    3. Terminate the SSH session and return to your host shell.

      $ exit

    2. Login as the admin user and decommission the edge-1 managed device, which corresponds to your first test VM.

      $ flightctl get device -o wide

    4. You must check the status field of the device to know that it has been decommissioned.

      $ UUID=1u63hvsgtua9tjjbmla9bgh5h2s77cb8esepphnf68m3ocf8qfng
$ flightctl get device $UUID -o json | jq -r '.status.lifecycle.status'
Decommissioning

  7. Delete the last device, also by using the CLI.

    1. Use the UUID shell variable that you set int the previous step to delete the device resource from RHEM.

      Notice that, when using the CLI, there is no confirmation for either decommissioning or deleting a managed device.

      $ flightctl delete device $UUID
Deletion request for device "1u63hvsgtua9tjjbmla9bgh5h2s77cb8esepphnf68m3ocf8qfng" completed

    2. Verify that your device list is now empty.

      $ flightctl get device -o wide
NAME    ALIAS   OWNER   SYSTEM  UPDATED APPLICATIONS    LABELS

  8. Verify that your test devices cannot communicate with their RHEM server anymore.

  9. Verify that the devices are still fully functional, by acessing their embedded web application.

    $ curl 127.0.0.1:8108
This would be the HTML and JavaScript code of a single-page web application.

You demonstrated that decommissioning and deleting managed devices on RHEM does not scrubs the devices. If they contained sentitive data, such as secrets to access corporate systems, that is still there, and you must implement a process for cleaning those devices of such data.
