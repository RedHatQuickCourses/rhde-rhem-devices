Decommission and Delete Edge Devices with RHEM

Estimated reading time: 2 minutes.

Objective

Make a device not managed by RHEM anymore and, optionally, delete the device resource.

Work in progress

Decommission versus delete operations

At some moment, you will have to remove your devices from RHEM management. Maybe a device is being sent to another location, for a different use. Maybe a device is just broken beyond repair, or lost.

RHEM provides a decommission operation, which requires the administrator role, and removes all labels of a device, thus removing it from any fleet. The device status is set to decomissioned, so it is hidden from regular device listings in the web UI.

The API resource of a decomissioned device is still stored, so you have a track record of device information. RHEM does not perform any scrubbing of the device: if there is sensitive data which must be securely deleted, you need a specialized tool for it.

You could have a fleet configured just for running a scrubbing tool, and move devices there before decomissioning them.

Decomissioned devices are not cleaned in any way: they retain the last system image, configurations, and applications that RHEM put there. Managed devices allways keep operating despite having access ro RHEM, including aftger being decomissioned.

At some moment, you will want to remove records of decomissioned devices, to keep your RHEM database nimble. The delete operation also requires the administrator role and it is unrecoverable.

After a device is deleted, it still contains a RHEM agent and an enrollment certificate, which could be used to re-enroll the device.

What’s next

The final activity of this course exercises decomissioning and deleting devices. After you complete it, stay tuned for the next course in the series, which teaches fleet and application management.
