= Lab: Enroll Edge Devices Using Late Binding

_Estimated reading time: *{time_estimate} minutes*._

Objective::

Provision and enroll devices on Edge Manager from an installation ISO.

== Before you begin

You need a _development machine_ running RHEL, to which you have unrestricted sudo access, and enough capacity to run a few VMs.
Your _development machine_ is already registered to Red Hat with a valid RHDE subscription.

You also need a _server machine_ running RHEL, to which you also have unrestricted sudo access, which was already configured with Edge Manager by the xref:ch2-deploy:s2-install-lab.adoc[previous lab].

Finally, you need a _private registry_ to store bootc container images for your managed edge devices.

Your _development machine_ requires internet access to download RPM packages and container images.

If you are using the course classroom, your _development machine_ is the `workstation` VM.
Log in as the user `student` with the password `student`.

== Instructions

You will build a minimal bootc container image, which contains the Edge Manager agent, and create an installable ISO which contains enrollment credentials.
Then you will use that ISO to provision a test VM, which stands in for a managed edge device, and verify that it attempts to enroll in your Edge Manager server.
Finally, you will approve that device enrollment request, and verify that you can use the same ISO, unchanged, to provision additional edge devices.

8. Enroll your second test VM on Edge Manager, but this time using its web interface.

.. Open a web browser and visit `\https://servera.lab.example.com`.

.. Log in as the `admin` user with the `redhat` password.

.. The overview page should indicate that you have one healthy managed device, with no applications.
You will see one blue circle followed by two green circles.

.. On the left panel, click menu:Devices[].
The beginning of the page indicates that there's one device pending approval.

.. Verify that the name of that device matches the UUID you got from the logs in your second test VM.

.. Scroll down and verify that the first test VM is listed as a managed device, on the same page.

.. Scroll back to the beginning of the page to approve your second test VM as a managed device.
+
Click on the menu:Untitled[] link on the menu:Alias[] column.
You can then see information about the device, such as its MAC address, if you must confirm its identity before approving it.

.. Return to the menu:Devices[] page and click on the menu:Approve[] link to the right.

.. Type `edge-2` as the alias of the device and click menu:Add label[] to show a text field where you will type the key and value for the next label.

.. Type `rhel=9.6` and press kbd:[Tab] to display the menu:Add label[] link again.
Repeat the process to add another label, and type `type=noapps`.
Feel free to add more labels, but be sure to include those two.

.. The warning _No fleet is matching the selected labels_ is expected and can be safely ignored. 
Click menu:Approve[].

.. The menu:Devices[] page now shows no devices pending approval, and shows the new managed device with an `Out-of-date` status.

.. Wait a few moments until the device synchronizes with Edge Manager and its status changes to `Up-to-date`.

9. Verify that, despite being provisioned from the same enrollment request data, each test VM retains its own identity and configuration from Edge Manager.

.. On the menu:Devices[] page, click on the menu:edge-2[] alias to enter the managed device details page of your second test VM.

.. Click menu:YAML[] to see the resource manifest of the device, as you would get from the `flightctl get device -o yaml` command.

.. Click menu:Terminal[] to get a shell on the managed device, as you would get from the `flightctl console` command.

.. Verify that the second test VM does not contain a `device.txt` file by using the `ls` command.

You now have two test VMs enrolled in Edge Manager as managed devices.
You also know how to identify each one and access these devices, to run troubleshooting commands, directly through Edge Manager.
Finally, you know that you can perform most actions by using either the CLI or the web interface.
One of the few exceptions, which requires using the CLI, is fetching enrollment request data.
